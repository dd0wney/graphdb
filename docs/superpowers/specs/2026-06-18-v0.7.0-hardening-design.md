# Design: v0.7.0 production-hardening track

**Date:** 2026-06-18
**Status:** Approved (brainstorming) → pending implementation plan
**Source:** `docs/ROADMAP_v1.md` blockers B1, B2, B3, B6 (the v0.7.0 "production hardening" cut).
**Scope decisions (from brainstorming):** B1 → tenant-scoped delete; B3 → measure-then-flip; B6 → docs/honest-signaling.

## Context

`v0.6.0` shipped. The ROADMAP defines v0.7.0 as the production-hardening release: four mostly-independent fixes that close correctness, durability, and honesty gaps gating a credible 1.0. This spec covers those four. Out of scope (v0.8+): hot backup/restore (B4), the API/format stability policy + JSON snapshot header (B5).

Each item is grounded in current code, verified against `main` @ `3885945`.

## B1 — tenant-scoped `DELETE /nodes`

**Problem.** `/nodes` is `requireAuth(withTenant(...))` but not admin-gated (`pkg/api/server.go:55`); its DELETE handler (`pkg/api/handlers_nodes.go:56`) calls the tenant-blind `graph.DeleteAllNodes()` (`pkg/storage/node_operations.go:848`), which clears every shard and every tenant index. Any authenticated tenant wipes all tenants' data.

**Design.** Add `func (gs *GraphStorage) DeleteAllNodesForTenant(tenantID string) error`:
- Enumerate the tenant's node IDs via the existing membership index (the same source `GetAllNodesForTenant` uses), snapshot the ID list, then delete each node through the **existing** `DeleteNodeForTenant` path (which cascades the node's edges).
- Reusing `DeleteNodeForTenant` is deliberate: that path already writes per-delete WAL records and already adds mmap tombstones, so (a) other tenants' data and the shared WAL/snapshot are untouched, and (b) it inherits the #423 mmap-correctness automatically — deleted nodes do not resurrect from the mmap base on reopen. A bulk/optimized path is a later optimization, not part of this fix.
- Switch the handler to `s.graph.DeleteAllNodesForTenant(getTenantFromContext(r))`.
- Keep the global `DeleteAllNodes()` as an internal/test utility; it is simply no longer reachable cross-tenant over HTTP.

**Tests (TDD).** Multi-tenant: populate tenants A and B (nodes + edges + a label); `DELETE /nodes` as tenant A. Assert A is empty (count 0, enumeration empty) and B is **byte-identical** to before (reuse the storage-layer fingerprint idea or a direct count+enumerate compare). Add a reopen check (close/reopen, B still intact, A still empty) to confirm the mmap-tombstone path. Cross-check the handler returns 200 and only touches the caller's tenant.

## B2 — graceful shutdown that drains

**Problem.** On SIGTERM (`cmd/server/main.go:499-516`) the handler builds a 30s timeout context, blocks on `<-ctx.Done()` (sleeps the full 30s), then `graph.Close()` + `os.Exit(0)`. No `http.Server.Shutdown()`; the listener is never stopped and in-flight requests are not drained.

**Design.**
- Promote the `http.Server` from a local in `Start()` (`pkg/api/server.go:253`) to a `Server` struct field (e.g. `httpServer *http.Server`), assigned before `ListenAndServe`.
- Add `func (s *Server) Shutdown(ctx context.Context) error` delegating to `s.httpServer.Shutdown(ctx)` (nil-safe if never started).
- In `cmd/server/main.go`: the signal goroutine calls `server.Shutdown(ctx)` (stops accepting, drains in-flight up to the deadline) then `graph.Close()`, and exits as soon as draining returns — not after a fixed 30s. `Start()`/`ListenAndServe` returns `http.ErrServerClosed` on graceful stop; main must treat that as a clean exit, NOT `os.Exit(1)`.

**Tests.** A focused server test: start on an ephemeral port, issue a request that's in-flight when `Shutdown(ctx)` is called, assert the in-flight request completes and `Shutdown` returns nil, and that a post-Shutdown request is refused. (Process-level SIGTERM wiring in `main.go` is covered by the manual smoke step in the plan; the drain semantics live in the testable `Server.Shutdown`.)

## B3 — measure then flip the BatchedWAL default

**Problem.** `DefaultStorageConfig` sets `EnableBatching: false` (`pkg/storage/storage.go:23`); `cmd/server` uses it unmodified → per-write fsync, serialized writes. `BatchedWAL` (group-commit, durability-preserving: a write's `Wait` blocks until its batch fsyncs) exists and is tested but never enabled in the binary. This is PERF HIGH-3.

**Design (measure-then-decide).**
1. Add a benchmark (e.g. `pkg/storage/bench_wal_flush_interval_test.go`, following the repo's existing bench-harness pattern) that measures write throughput and per-write latency for: per-write-fsync baseline, and BatchedWAL at FlushInterval ∈ {1ms, 2ms, 5ms, 10ms}, under a representative concurrent-writer count.
2. **Controller runs the benchmark, records the numbers, and picks the knee** (the interval that captures most of the throughput gain without an unacceptable p99 latency rise). The chosen value is recorded in the plan/commit.
3. Set `DefaultStorageConfig` `EnableBatching: true` with the tuned `FlushInterval`; verify the config plumbs through to the server's storage init.

**Durability invariant.** Flipping must not weaken durability: a write still returns only after its batch is fsynced. The existing `pkg/storage/integration_wal_batched_test.go` + crash-recovery suite (`crash_recovery_test.go`) must stay green with batching on by default.

**Tests.** The benchmark (numbers captured, not asserted). Plus: confirm the crash-recovery + batched-integration suites pass with the new default, and a small assertion that `DefaultStorageConfig()` now returns `EnableBatching: true` with the chosen interval.

## B6 — honest single-node signaling

**Problem.** `pkg/cluster/` (~2800 LOC) is real but unwired (`pkg/cluster/doc.go` says no replication append path; `cmd/server`/`pkg/api` import none of it). Shipping it implies HA that doesn't exist; cluster Prometheus gauges report static zeros.

**Design (docs + signaling, no code-path change).**
- Strengthen `pkg/cluster/doc.go` to a prominent `EXPERIMENTAL — NOT WIRED — single-node only` header stating it is not connected to the graph store and not in the server binary's path.
- State "single-node only (clustering is experimental/unwired)" in `README.md` and `docs/CAPABILITIES_2026-05-10.md`'s cluster entry.
- Add a guard test (`pkg/cluster` or a top-level arch test) asserting `cmd/server` / `pkg/api` do not import `pkg/cluster`, so the "unwired" claim can't silently regress.
- (Optional, minor) note in `pkg/metrics/init_cluster.go` that the cluster gauges are inert in single-node mode. Keep this light per the chosen scope.

**Tests.** The import-guard test. Docs changes need no test.

## Track structure

Branch `feat/v0.7.0-hardening`. Four independent tasks (B1, B2, B3, B6), each its own commit, executed subagent-driven with per-task review + a final whole-branch review. B3's measurement is controller-run (the benchmark task lands the harness; the controller runs it and supplies the interval for the flip). At finish, decide per-item PRs vs. one track PR.

## Out of scope (YAGNI / later releases)

- Hot backup/restore endpoint (ROADMAP B4 → v0.8.0).
- API/format stability policy + JSON snapshot versioned header (ROADMAP B5 → v0.8/v1.0).
- Any bulk/optimized tenant-delete path (B1 reuses the per-node cascade; optimize later if measured slow).
- Wiring or building the cluster (B6 is signaling only).

## Risks

- **B1 performance on huge tenants:** the per-node delete loop is O(tenant size). Acceptable for the wiki-graph reload use case; flagged for a future bulk path if a consumer hits it. Not a correctness risk.
- **B3 latency regression:** mitigated by measuring before flipping and choosing the knee; the durability invariant is preserved by `Wait`-blocks-until-flushed.
- **B2 shutdown wiring:** `main.go` must distinguish `http.ErrServerClosed` (clean) from real errors, or a graceful stop would exit non-zero.
