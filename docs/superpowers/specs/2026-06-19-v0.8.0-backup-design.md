# Design: v0.8.0 — hot backup endpoint (ROADMAP B4)

**Date:** 2026-06-19
**Status:** Approved (brainstorming) → pending implementation plan
**Source:** `docs/ROADMAP_v1.md` blocker B4 (v0.8.0 operability). Scope decision (brainstorming): **download backup + documented offline restore**.

## Context

graphdb has no OSS backup story beyond a cold `tar` of a stopped volume; hot backup ships only in the enterprise `r2-backup` plugin. v0.8.0 closes that with a built-in, admin-only hot-backup endpoint that streams a self-contained archive, plus a documented offline restore. This works in containers where the operator can't reach the filesystem.

**Verified against `main` @ `b6eefef`:**
- `Snapshot()` (`pkg/storage/persistence.go:24`) is live-safe — RLock for a pointer-clone of indexes, marshals after releasing; writes `snapshot.json` (`GSNP`-enveloped) or `snapshot.mmap`. It does NOT truncate the WAL (that's `CompactWAL`).
- Load is **construction-time only** (`loadFromDisk`/`loadFromDiskMmap` in `NewGraphStorageWithConfig`, `storage.go:144-159`) + `replayWAL`. There is no hot in-place restore path — restore = stop, replace files, reopen.
- A complete backup is, under `dataDir`: the snapshot file + `wal/` + `auth/` (users + apikeys — **password hashes**) + `lsa/` (+ `edgestore/` if `UseDiskBackedEdges`). Vector indexes are NOT serialized; they rebuild from nodes after WAL replay.
- `requireAdmin` (`pkg/api/middleware_auth.go:18`), route registration (`pkg/api/server.go` `registerRoutes`), `respondJSON`/`respondError` (`pkg/api/server_helpers.go:116,124`), `s.graph` (`*storage.GraphStorage`) and `s.dataDir` (`server_types.go:73`) are all available to mirror.

## Component design

### 1. The backup handler (`pkg/api/handlers_backup.go`, new)

`func (s *Server) handleBackup(w http.ResponseWriter, r *http.Request)`:
1. Method gate: non-`POST` → 405. (POST because it has a side effect — it writes a snapshot.)
2. `s.graph.Snapshot()` — flush a consistent point. On error → 500 via `respondError`.
3. Set headers: `Content-Type: application/gzip`, `Content-Disposition: attachment; filename="graphdb-backup-<UTC RFC3339, ':'→'-'>.tar.gz"`.
4. Stream: wrap `w` in `gzip.NewWriter` → `tar.NewWriter`. Write a `manifest.json` entry first (see §3), then walk `s.dataDir` and add each eligible file.
5. On a mid-stream write error, log it; the partial response is already committed (status 200 sent), so the client detects truncation via the gzip/tar EOF — documented behavior.

**File selection** (a helper, e.g. `backupEntries(dataDir) ([]string, error)`):
- Include: the snapshot file present (`snapshot.json` xor `snapshot.mmap`), and everything under `wal/`, `auth/`, `lsa/`, `edgestore/`.
- Exclude: `*.tmp` (transient snapshot temp files), and the archive must store paths **relative to `dataDir`** so extraction reconstructs the same layout.
- Read each file with a fresh `os.Open`; a WAL segment being appended mid-read yields at most a partial trailing record — tolerated by WAL replay on restore (stops at the last valid record). This is the documented consistency guarantee: snapshot-consistent + best-effort-current WAL.

### 2. Route registration (`pkg/api/server.go`)

In `registerRoutes`, alongside the other `/admin/*` routes:
```go
mux.HandleFunc("/admin/backup", s.requireAdmin(s.handleBackup))
```

### 3. Manifest (`manifest.json`, first tar entry)

A small JSON struct written into the archive root:
```go
type backupManifest struct {
    GraphdbVersion string   `json:"graphdb_version"` // main.Version, threaded in or "dev"
    CreatedAtUTC   string   `json:"created_at_utc"`  // RFC3339
    SnapshotMode   string   `json:"snapshot_mode"`   // "json" | "mmap"
    Files          []string `json:"files"`           // relative paths included
}
```
Purpose: human-readable provenance + a restore-time sanity check. The version string is whatever the server was built with; if not readily threadable into `pkg/api`, record `"unknown"` rather than inventing a value (note for the plan).

### 4. Restore (offline) — `docs/BACKUP_RESTORE.md` (new) + `DEPLOYMENT_GUIDE.md` pointer

Document the only supported restore (construction-time load):
1. Stop the server.
2. Clear/replace the target `dataDir` (back up the existing one first).
3. `tar xzf graphdb-backup-<ts>.tar.gz -C <dataDir>`.
4. Start the server — `loadFromDisk[Mmap]` + WAL replay reconstruct the graph; vector indexes rebuild from nodes.
Plus a **security note**: the archive contains `auth/` password hashes and all tenants' data — store/transmit it like a credentials file; the endpoint is admin-only and should be used over TLS.

## Testing

- **`handleBackup` unit test** (`pkg/api/handlers_backup_test.go`): seed a store via the existing `setupTestServer` helper, write a few nodes/edges, `POST /admin/backup`, gunzip+untar the response, assert entries include the snapshot file, a `wal/` entry, `auth/users.json` (or the auth files), and a parseable `manifest.json` with the expected mode + non-empty `Files`.
- **Round-trip test**: extract the archive into a fresh temp `dataDir`, `NewGraphStorage` on it, and assert the restored graph matches the source (node/edge counts + a per-tenant fingerprint comparison, reusing the storage-test fingerprint idea). This is the real proof that a backup restores.
- **Admin-gating test**: a non-admin token → 403; an unauthenticated request → 401. Mirror an existing `/admin/*` gating test.
- **Method test**: `GET /admin/backup` → 405.

## Risks

- **WAL-tail rebuild during stream**: mitigated — replay tolerates a partial trailing record; the guarantee is "snapshot-consistent + best-effort current WAL," documented.
- **Large stores**: streaming (no temp file) bounds memory, but a backup of a multi-GB store holds the snapshot RLock only for the clone, then streams — acceptable. No size cap in v0.8.0 (note as a possible follow-up).
- **Sensitive payload**: `auth/` hashes in the archive — admin-only + TLS + docs warning. Not encrypting the archive in v0.8.0 (the operator owns transport security); archive-encryption is a possible follow-up.

## Out of scope (YAGNI / later)

- Hot in-place restore endpoint (the store loads only at construction; a live swap is a separate, riskier track).
- Scheduling, retention, incremental/differential backups, remote targets (R2/S3) — these are the enterprise `r2-backup` plugin's domain.
- Archive encryption / signing.
- B5b (the written API/format stability policy) — stays a v1.0.0 item.
