# A8.1 spike — replication standalone-binary disposition

**Status**: spike output. Decision needed. **Do not pre-judge**.
**Predecessor**: `docs/A8_REPLICATION_TENANCY_DESIGN.md` §5 (`Q5: Standalone binaries — fail-closed startup gate, not just a comment`, Q7 commit 4, line 232 follow-up).
**Source request**: `docs/NEXT_STEPS_2026-05-10.md` §"A8.1" + §"Sequencing graph" — A8.1 is the new head of the critical path after F3 closed (2026-05-12, PRs #104/#107/#111/#114/#122/#124).

---

## TL;DR

Three options on the table; spike surfaces them but does not pick. **The directive framed this as a binary (rebuild vs delete); the survey below shows it is a trilemma**.

- **Option A — Rebuild standalone replication on `cmd/server`**: net-new work, not refactoring. `cmd/server` does not currently import `pkg/replication`. Multi-PR, touches audit middleware + WAL-stream tenancy.
- **Option B — Delete the legacy binaries + the `pkg/admin/upgrade*` orphan + rewrite docs to say "single-node only"**: smallest code surface, largest user-facing doc churn. Honest about the planning-doc's stated "single-node by design" trajectory.
- **Option C — Delete binaries + retain `pkg/replication` library for an eventual rebuild** (defer the deployment story): keeps the library intact for a future `cmd/server`-native replication rebuild without committing to it now.

Recommendation: see §6. Final call is the user's.

## 1. Context

`A8_REPLICATION_TENANCY_DESIGN.md` Q5 placed `cmd/graphdb-{primary,replica}` (+ `nng` variants) behind a fail-closed `GRAPHDB_LEGACY_BINARY=1` env gate (PR #47). Q5 explicitly called the gate **a holding action, not a fix**. Q7 commit 4 deferred the fix to a follow-up: "deprecate or rebuild standalone replication binaries on top of `cmd/server` infrastructure" (line 232).

That follow-up is A8.1. The planning doc (`NEXT_STEPS_2026-05-10.md` §"A8.1") states the acceptance criterion as:

> `GRAPHDB_LEGACY_BINARY` becomes either (a) a removed env var, or (b) gates only a thin shim that delegates to `cmd/server` plumbing.

(a) corresponds to **Option B/C** in this spike; (b) corresponds to **Option A**.

## 2. Survey — what is actually in the tree

### 2.1 The four legacy binaries (`cmd/graphdb-{,nng-}{primary,replica}`)

| Binary | LOC | What it has |
|---|---|---|
| `cmd/graphdb-primary/main.go` | 127 | TCP transport. HTTP: `/health`, `/stats`, `/replication/status`, **`POST /nodes` (unauth, default-tenant via `graph.CreateNode`)**. |
| `cmd/graphdb-replica/main.go` + `server_test.go` | 131 + 75 | TCP transport. Read-only HTTP: `/health`, `/stats`, `/replication/status`. **A8.2 (PR #81) already removed `/nodes`** + extracted `buildHTTPHandler` + pinned the route set with `TestBuildHTTPHandler_A82_NoNodesRoute`. |
| `cmd/graphdb-nng-primary/main.go` | 165 | NNG transport (`//go:build nng`). Same HTTP surface as TCP primary, plus `--add-dc` datacenter-link CLI. Same `POST /nodes` hole. |
| `cmd/graphdb-nng-replica/main.go` + `server_test.go` | 154 + 75 | NNG transport (`//go:build nng`). Same shape as `graphdb-replica` post-A8.2. |
| **Total** | **727** | (incl. tests). Replicas are already mostly skeletons; primaries still have the live `POST /nodes` tenancy hole. |

All four binaries `log.Fatalf` at startup unless `GRAPHDB_LEGACY_BINARY=1` is set. Messages explicitly say "Use cmd/server."

### 2.2 `cmd/server` — current replication story

**`cmd/server/main.go` does not import `pkg/replication`.** Production deploy path is single-node by design — consistent with `NEXT_STEPS_2026-05-10.md` line 270:

> Single-node assumption baked in. Replication is primary/replica (A8 design), not sharded — write throughput can't scale beyond one box. Any horizontal-scale story (sharded write path, consensus, distributed query) is a multi-quarter scope.

So **Option A is not "wire `cmd/server` to the existing replication"** — it's "design a tenancy-aware HTTP-replication path inside `cmd/server` and migrate the library to it." Net-new work.

### 2.3 `pkg/replication/` — the shared library

30+ files, substantial infrastructure: TCP/NNG transports, primary cluster, replica connection, heartbeats, health surveyor, write forwarding, protocol messages, integration tests. Consumed by:

- The four legacy binaries (Section 2.1).
- **`pkg/admin/upgrade*.go` — 1,295 LOC including 789 LOC of tests.** Implements failover orchestration: `upgrade_promote.go:100-157` instantiates `replication.ReplicationManager` and `replication.ReplicaNode` for the replica-to-primary promotion path.

**`pkg/admin/upgrade*` has zero external consumers** — `grep -rn "admin.NewUpgradeManager"` returns empty across the entire repo. It is dead code at the call-graph level. Either: (a) the wiring was removed but the implementation was not; (b) it was scaffolded for a future operator-API that was never built. Either way, deleting the legacy binaries does not break `pkg/admin/upgrade*`'s callers, because **there are no callers**.

### 2.4 Docs surface — the contradiction the user must see

Six docs reference the legacy binaries. By LOC + audience:

| Doc | LOC | Audience | Says |
|---|---|---|---|
| `docs/PRODUCTION_QUICKSTART.md` | **811** | **External users** | Step-by-step deployment of `graphdb-primary` + `graphdb-replica` for production. **Zero mention of `GRAPHDB_LEGACY_BINARY` or audit A8.** Self-contradicts the audit's "not for production" verdict. |
| `docs/UPGRADE_GUIDE.md` | 512 | External users (operators) | Upgrade + failover procedures for the legacy binaries. **Has the A8 disclaimer at the top** (lines 4-11) noting they refuse to start without `GRAPHDB_LEGACY_BINARY=1`. |
| `docs/INTEGRATION_GUIDE.md` | 723 | External users | Uses `cluso-graphdb-primary` as a Digital Ocean droplet hostname; lighter reference. |
| `docs/MONITORING_SETUP.md` | 651 | Operators | One Prometheus label reference (`instance: 'graphdb-primary'`). |
| `docs/CAPABILITIES_2026-05-10.md` | — | Internal | Capability catalog entry; informational. |
| `README.md` line 284 | — | External users | Names the `GRAPHDB_LEGACY_BINARY` gate. |

**The contradiction**: `PRODUCTION_QUICKSTART.md` is the file an external user reads first. It points them at the binaries the audit says are unsafe for production, without mentioning the fail-closed gate. Either the quickstart is stale and never reconciled with A8, or there is an implicit understanding that "production" here means "opt-in legacy mode." Both interpretations are uncomfortable; the spike does not resolve which is correct.

## 3. Option A — Rebuild standalone replication on `cmd/server`

### Scope (rough)

1. Design `cmd/server` HTTP-replication endpoints (primary stream + replica subscribe + write-forward). Tenancy-aware throughout — same `*ForTenant` discipline as the rest of `cmd/server`.
2. Migrate `pkg/replication/`'s transport layer to use `cmd/server`'s auth + tenant middleware (or replace it with an HTTP-native layer reusing `cmd/server`'s plumbing).
3. Update `pkg/admin/upgrade*` to use the new path (or accept its deletion — see Option B/C).
4. Decide: keep `pkg/replication/nng_*` and the NNG-PUB/SUB transport, or drop NNG in favor of HTTP-only (the H1 closure already removed ZMQ; dropping NNG would complete the protocol-rationalization).
5. Rewrite `PRODUCTION_QUICKSTART.md` against the new `cmd/server` topology.
6. Delete the four legacy binaries.
7. Audit-regression row for tenancy-preservation through the new path.

### Cost

Multi-quarter. The A8 design doc itself §3 names "end-to-end socket round-trip test" and "tenant-aware ACL on replication forwarding" as deferred items already; both reactivate here. The "design tenancy-aware HTTP replication" step alone is design-spike-sized work.

### Gain

- A `cmd/server`-native replication story consistent with the rest of the codebase.
- `GRAPHDB_LEGACY_BINARY` gate retired.
- Production-quickstart claims become honest.

### Risk

- Multi-quarter scope was explicitly out-of-scope per `NEXT_STEPS_2026-05-10.md` planning principles ("Capacity assumption: ~2-3 PRs/week").
- Re-opens A8's deferred items (Q6 §6 end-to-end test, §6 tenant-aware ACL on forwarding).
- Implies a horizontal-scale commitment the planning doc explicitly declines to make.

## 4. Option B — Delete + rewrite docs to single-node honest

### Scope

1. Delete `cmd/graphdb-{,nng-}{primary,replica}/` (4 directories, ~727 LOC).
2. Delete `pkg/admin/upgrade*.go` (~1,295 LOC, zero external consumers).
3. Delete or shrink `pkg/replication/` depending on what remains (probably most of it goes — see open question §7 Q1).
4. Rewrite `docs/PRODUCTION_QUICKSTART.md` against `cmd/server` only. Likely delete `docs/UPGRADE_GUIDE.md` (its entire surface is the legacy binaries).
5. Update `README.md` line 284 to remove the `GRAPHDB_LEGACY_BINARY` reference.
6. Update `docs/MONITORING_SETUP.md`, `docs/INTEGRATION_GUIDE.md`, `docs/CAPABILITIES_2026-05-10.md` to match.
7. Update `pkg/api/audit_regression_test.go` reference map — A8.1 closed by deletion.
8. Update `docs/NEXT_STEPS_2026-05-10.md` Track A to mark A8.1 done.

### Cost

Mechanical. Estimated 3-5 PRs (one per logical chunk: binaries+admin, library shrink, quickstart rewrite, observability/README cleanup, planning-doc update).

### Gain

- `GRAPHDB_LEGACY_BINARY` gate retired by removal.
- Production-quickstart claims become honest (single-node by design, matches `NEXT_STEPS_2026-05-10.md` L270).
- ~2,000 LOC removed (binaries + admin orphan, before counting `pkg/replication/` shrink).
- Closes the audit-doc-vs-quickstart contradiction surfaced in §2.4.

### Risk

- **Unknown external users.** If anyone is running the legacy binaries with `GRAPHDB_LEGACY_BINARY=1` based on `PRODUCTION_QUICKSTART.md`, deletion is a breaking change. Surfacing this is the user's call — the spike cannot poll users.
- Exits the multi-node deployment story until a future rebuild. The planning doc's L270 framing accepts this; the user must confirm.

---

## 5. Option C — Delete binaries + retain `pkg/replication` library

### Scope

Like Option B, but step 3 becomes "keep `pkg/replication/` intact for a future rebuild" and step 4 keeps a thin pointer in docs ("multi-node replication library exists; CLI binaries removed pending a `cmd/server`-native deployment story").

### Cost

Marginally less than Option B (no library shrink, no aggressive doc rewrite). Estimated 2-3 PRs.

### Gain

- All of Option B's gains except the library-LOC reduction.
- Preserves replication-library investment as an option-value for a future deployment-story rebuild.

### Risk

- Carries dead-but-documented infrastructure. Future agents will wonder why `pkg/replication/` exists with no callers.
- Closing-but-not-removing is the kind of "documented as deprecated, structurally still present" pattern the audit said it wanted to *avoid* repeating (the A8 fail-closed gate is itself an example of that holding-action pathology).

## 6. Decision criteria

The choice between A, B, C depends on three user-level decisions:

1. **Commercial offering shape** (open `NEXT_STEPS_2026-05-10.md` §"Decision points" carry-forward — *Compliance API: REST-only or also GraphQL* was the analogous item, now RESOLVED). Hosted offering or enterprise self-serve makes Option A more defensible. Open-source-first with paid support pulls toward B/C.
2. **External-user exposure to the legacy binaries.** Anyone deploying off `PRODUCTION_QUICKSTART.md` today is in the affected set. The spike cannot resolve this; the user (or a customer poll) must.
3. **Horizontal-scale commitment**. The planning doc declines this; if the user wants to re-open it, Option A is back on the table.

A recommendation pre-judged by the spike would be **Option B**, on grounds that: (a) `cmd/server`'s "single-node by design" is the current architectural truth; (b) the production-quickstart contradiction is already a fairness-to-users issue; (c) `pkg/admin/upgrade*` is dead code already. **But the spike defers to the user**, because risks (1) and (2) above are inputs the spike cannot see.

---

## 7. Open questions for the user

1. **Does `pkg/replication/` retain any value if all four binaries delete?** If `pkg/admin/upgrade*` also deletes, every consumer is gone. Option B says delete; Option C says keep for future rebuild. The honest framing is: are we committing to *eventually* having multi-node, or are we accepting single-node as the architectural endpoint?
2. **External-user exposure**: do we know of anyone running `graphdb-primary`/`graphdb-replica` in production? If not, deletion is safe. If unknown, the spike's recommendation should include a deprecation window with the gate.
3. **`PRODUCTION_QUICKSTART.md` provenance**: who wrote it, when, and was it ever reconciled with the A8 audit? If it's stale (the most likely interpretation), confirming that determines whether deletion is a quiet doc-correction or a contract break.
4. **NNG vs HTTP transport**: if Option A is picked, do we keep the NNG PUB/SUB layer or move to HTTP-only? (H1 already removed ZMQ for the same protocol-rationalization argument.)
