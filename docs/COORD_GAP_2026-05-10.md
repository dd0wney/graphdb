## What the codebase actually exposes (as of fc0040b)

- **`cmd/graphdb`** is a demo program. No flags, no HTTP, runs a hard-coded
  trust-network example and exits. Not a server.
- **`cmd/server`** is the real HTTP server. Flags: `--port`, `--data` only.
  Auth comes from license + API key env vars, not a `--jwt-secret` CLI flag.
- **`cmd/license-server`** is a Stripe-backed license issuance service
  (flags: `--port`, `--database-url`, `--data`, `--stripe-key`,
  `--webhook-secret`). It is NOT a `jwt issue` CLI; there is no
  `license-server issue --subject ... --tenant ...` subcommand.
- **`cmd/graphdb-admin`** has only `security` (init / rotate-keys /
  audit-export / health / key-info) and `help` / `version`. No schema-
  bootstrap subcommands.
- **REST routes** registered in `pkg/api/server.go`:
  - `/v1/embeddings`, `/v1/retrieve` are the only `/v1/` paths.
  - Node / edge CRUD lives at the unversioned root: `/nodes`, `/nodes/`,
    `/nodes/batch`, `/edges`, `/edges/`, `/edges/batch`.
  - No `/v1/constraints/...`, `/v1/property-indexes`, `/v1/batch`, or
    `/v1/nodes/by-property`. None of these endpoints exist.
- **GraphQL `/graphql`** is registered and `pkg/graphql/edges_schema.go`
  exposes `createNode` / `updateNode` / `deleteNode` / `createEdge` /
  `updateEdge` / `deleteEdge` mutations. So a GraphQL-based work-claim is
  feasible.
- **Storage primitives exist**: `pkg/storage.PropertyIndex`,
  `pkg/constraints.UniquenessViolation` (+ `pkg/constraints/uniqueness.go`).
  The primitives are not wired through the API.

## Gaps grouped by severity

### Wiring gaps (existing primitive, no API surface)

1. **Property indexes** — `pkg/storage` already maintains them; no HTTP/
   GraphQL surface to declare one at runtime. Coord schema bootstrap
   (Task.id, Agent.id) needs this.
2. **Uniqueness constraints** — `pkg/constraints/uniqueness.go` exists; no
   API surface to declare one. Coord schema bootstrap needs this for "at
   most one active Claim per Task."

### Correctness gap (atomicity)

3. **No multi-mutation transaction primitive.** `work-claim` claims
   atomicity via `POST /v1/batch`, but `/v1/batch` doesn't exist. Even if
   we route through GraphQL or `/nodes/batch`+`/edges/batch`, the
   Claim-node creation and the `:HOLDS` / `:FOR` edge creations are
   separate operations. Two agents can each create a Claim node before
   either gets to the edge — both think they own the task.
   - Note: `/nodes/batch` exists (line 52 of server.go) but its semantics
     (single-type batch) almost certainly don't extend to "create node +
     edges atomically." Worth a closer look before assuming.

### Doc / route-prefix gaps (cosmetic)

4. **`/v1/` prefix mismatch.** Skills say `/v1/nodes`, server registers
   `/nodes`. Trivial doc fix if we go option A.
5. **License-server CLI shape.** The doc's `license-server issue …`
   command doesn't exist; would need either a JWT-issuance subcommand or
   a different token path.

## Implications for the three options

- **A. Align skills to the existing API**: viable via GraphQL. Skill bash
  blocks rewrite to `curl /graphql` with mutation strings. **Atomicity
  gap (#3) remains** — uniqueness must be advisory ("look before leap")
  with an accepted race window, OR we add server-side uniqueness check
  inside the `createNode` resolver for `Claim` specifically. The doc work
  is small; the atomicity story needs an explicit decision.

- **B. Build the missing surface**: implement `/v1/constraints/uniqueness`,
  `/v1/property-indexes`, `/v1/batch`, `/v1/nodes/by-property`. Several
  PRs. This is the "real API feedback" `NEXT_SESSION_PROMPT.md`
  anticipated. Adds value beyond coord — these primitives are useful for
  any caller.

- **C. Defer A8.2, capture findings, pick a different task**: this doc
  becomes a planning-doc entry; PR #78 stays scaffolding; the next coord-
  deploy attempt is gated on (some of) B landing.

## Things to verify before committing to any option

- Does `/nodes/batch` actually do multi-mutation, or just bulk-create
  homogeneous nodes? Read `handlers_nodes.go:handleBatchNodes`.
- Does the server's auth path accept JWTs at all, or is it API-key-first?
  The work-claim skill assumes `Authorization: Bearer <jwt>` works.
- Is there an `apikey` issuance flow that's simpler than JWT for the
  coord-instance use case? `handlers_apikeys.go` exists.
